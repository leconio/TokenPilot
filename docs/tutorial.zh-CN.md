# 第一个应用完整教程

[English](tutorial.md)

本教程从空白管理页面开始，完成一个应用、一次带失败后改用能力的直接模型调用、不含内容的用量上报、模型花费与 AI Unit 统计，以及用户额度。Node 和 Python 示例可以直接在 macOS 运行，不需要本机容器环境。

## 1. 完成首次设置并创建应用

打开管理页面并完成首次设置。TokenPilot 会创建管理员和第一个应用，同时生成一把具备 SDK 与 Connector 所需权限的应用接入密钥。密钥原文只显示一次，请立即保存。

需要第二个应用时，从应用切换处创建即可。不同应用不会共享用户、报表、字段、额度和分流发布状态。

在一次性测试环境中，可以向已经存在的应用写入两个不含密钥的调用连接、两个真实模型和一个带备用模型的虚拟模型：

```bash
TOKENPILOT_EXAMPLE_APPLICATION_SLUG='my-application' \
  pnpm --filter @tokenpilot/db db:seed:example
```

普通 `db:seed` 不会添加业务数据。示例命令只保存环境变量名称，不保存密钥值，并且可以重复执行。

## 2. 定义允许上报的字段

打开**字段**，只添加确实需要搜索或制作报表的业务维度：

| 范围 | 字段            | 类型  |
| ---- | --------------- | ----- |
| 事件 | `next_action`   | 文本  |
| 事件 | `voice_enabled` | 是/否 |
| 用户 | `parse_context` | 文本  |
| 用户 | `voice_type`    | 文本  |

敏感维度应标记为敏感。未定义字段或类型错误的值会被拒绝，保证搜索和报表含义稳定。提示词和模型回复不属于自定义字段，不应上报。

## 3. 添加调用连接和真实模型

打开**模型 → 调用连接**，添加一个“兼容 OpenAI”连接。填写服务地址和保存密钥的环境变量名称，例如 `OPENAI_API_KEY`；真正的密钥只放在 SDK 进程的环境变量中。运行环境准备好这个变量后，可以使用**检查连接**验证。

打开**模型 → 真实模型**，在这个连接下添加模型。请求名称必须与服务商要求的模型标识完全一致。需要验证失败后改用时，再添加第二个真实模型。

进入每个真实模型并发布 AI Unit 换算率。SDK 或 Connector 已经上报的实际金额不需要配置价格；调用端无法提供成本时，再添加按顺序匹配的备用规则。规则可以使用内置字段或已定义的自定义属性，并组合每次调用固定金额与输入、输出、缓存、推理、图片、语音或自定义用量的每单位金额。

## 4. 创建并发布虚拟模型

打开**虚拟模型**，创建业务代码长期使用的简短名称，例如 `support-chat`。先添加主要真实模型，再添加备用真实模型。可选条件可以按时间、用户、用户组或临时切换选择候选项。使用**模拟**输入代表性用户和输入类型，解决全部校验提示后再发布。

目标运行程序确认应用了完全相同的配置版本后，这次发布才算生效。被拒绝的发布会同时展示所有原因，最后一次成功配置仍会继续使用。

## 5. 通过 SDK 调用虚拟模型

先构建 Node SDK，再提供控制服务地址、应用密钥、应用标识，以及调用连接中指定的服务商密钥：

```bash
pnpm --filter @tokenpilot/node-sdk build
AI_CONTROL_URL=http://127.0.0.1:4000 \
AI_CONTROL_POLICY_API_KEY='一次显示的应用密钥' \
AI_CONTROL_APPLICATION_SLUG='my-application' \
OPENAI_API_KEY='服务商密钥' \
node examples/node-sdk/app.mjs
```

Python SDK 使用相同的虚拟模型规则：

```bash
PYTHONPATH=sdks/python/src \
AI_CONTROL_URL=http://127.0.0.1:4000 \
AI_CONTROL_POLICY_API_KEY='一次显示的应用密钥' \
AI_CONTROL_APPLICATION_SLUG='my-application' \
OPENAI_API_KEY='服务商密钥' \
uv run --project sdks/python python examples/python-sdk/app.py
```

SDK 会下载已发布候选项；启用额度限制时先预留 AI Unit；调用真实模型；只在规则允许时改用下一候选项；完成后按实际用量结算，并为每次尝试排队上报一条不含内容的事件。完整代码与自定义调用方式参见[接入指南](integration.zh-CN.md)。

已有应用继续使用 LiteLLM 时，运行单独的 [`litellm-local`](../examples/litellm-local/README.md) 示例即可。它使用相同的上报与虚拟模型发布逻辑，服务商密钥仍由 LiteLLM 自己管理。

## 6. 验证用户和统计

打开**用户**。上报中的 `user_id` 已经自动出现在当前应用，`display_user` 会作为易读名称显示。后台也可以只填写 `user_id` 手工新增用户。

继续检查**用量明细**、**模型花费**和 **AI Unit**。组合示例用户、虚拟模型、真实模型、尝试结果和自定义字段进行筛选。详情应展示 Token、完整改用链路、分流原因、花费、AI Unit 和类型明确的属性，但不能出现提示词或模型回复。保存当前分析后，可以加入当前应用首页。

## 7. 验证额度控制

为示例用户分配 AI Unit 额度。启用强制限制后，SDK 会在发出服务商请求前预留估算值，成功后按最终换算结果结算；请求无法开始时则释放预留。重置额度会追加带审计的调整记录，不删除历史。拉闸后，新的运行配置生效时会阻止后续调用。
