# 项目详细介绍

[English](guide.md)

## TokenPilot 解决什么问题

一个 AI 应用通常从单个模型和一个 API 密钥开始。业务增长以后，模型服务商后台往往无法准确回答下面这些问题：

- Token 是哪个产品功能、用户或调用路径消耗的？
- 一次调用最后失败了，但中间尝试已经产生了多少模型成本？
- 服务商和模型版本经常变化，业务代码应该使用什么稳定名称？
- 不同模型的价格差异，怎样换算成统一、容易解释的产品用量？
- 某个用户还剩多少用量，并发请求时怎样避免重复消耗同一份额度？

TokenPilot 把这些工作放在模型请求链路之外。LiteLLM 继续管理服务商密钥、重试和模型流量；TokenPilot 在一次尝试结束后接收不含内容的事件，可靠地完成处理，再把精简的调用策略发布给可信 SDK 缓存。

## 四个主要功能区

### 数据统计

管理页面可以查看模型调用、Token 明细、延迟、错误、模型花费、AIU 用量和配置覆盖缺口。筛选条件可以组合时间、用户、虚拟模型、真实模型、服务商、应用功能和调用结果。统计始终读取 ClickHouse，不会在故障时偷偷切换到另一套口径。

### 模型配置与分流

**模型**就是 LiteLLM 使用的真实模型名称，模型花费和 AIU 换算率都直接绑定它。**虚拟模型**是业务应用使用的稳定名称，例如 `fast-text` 或 `deep-reasoning`。它内部包含默认模型、失败顺序、时间条件和临时切换。每次发布都会生成不可变的配置版本。

### 服务商成本与 AI Unit

服务商成本表示模型服务实际产生的花费；AI Unit 表示产品自己定义的用量。两者分别计算，并保存当次决定实际使用的已发布单价快照。缺少单价的调用会明确显示为“未定价”或“未计算”，绝不会按零处理。

### 用户额度

AI Unit 可以只统计，也可以在接近额度时提醒，或者强制限制。强制限制使用带签名的预留记录，避免并发请求重复使用同一份剩余额度。成功调用会结算预留，取消的调用会释放预留，长期未完成的预留会自动过期。

## TokenPilot 明确不做的事情

- 不代理或改写模型请求。
- 不保存模型服务商密钥。
- 不需要提示词或模型回复。
- 不替代业务自己的客户和收款系统。
- 不支持缺少任一数据组件的简化模式。

## 推荐接入顺序

1. 同时部署 TokenPilot 和三个数据组件。
2. 先以“只统计”模式连接 LiteLLM。
3. 录入真实请求使用的每个 LiteLLM 模型名称。
4. 发布完整的服务商成本和 AI Unit 单价。
5. 检查覆盖率和统计结果。
6. 创建稳定的虚拟模型名称并发布分流配置。
7. 先用“只统计”模式设置用户额度。
8. 只有在模型身份和单价覆盖完整后，再启用提醒或强制限制。

这个顺序能让首次接入保持可观察、可调整，同时不会把缺失数据伪装成正确结果。
