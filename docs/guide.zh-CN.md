# 项目详细介绍

[English](guide.md)

## 用途

服务商后台通常只提供账号级用量。业务应用还需要按用户、功能和每次模型尝试查看数据，也可能需要稳定的模型名称、自定义用量单位和用户额度。TokenPilot 统一保存这些规则和结果。

应用可以使用 Node SDK、Python SDK、LiteLLM Connector 或手动上报。SDK 和 Connector 不会上报提示词与回复内容。网络短暂中断时，用量先留在本地队列；控制服务暂时不可用时，调用端继续使用最后一份有效分流配置。

## 主要部分

### 应用与用户

每个应用都有自己的首页、用户、字段、密钥、报表、额度和分流配置。调用必须提供 `user_id`，`display_user` 是页面显示的可选名称。用户可以随第一条用量自动创建，也可以手工添加。自定义字段需要先定义类型，可记录 `next_action`、`parse_context` 或语音模式等业务值。

### 调用连接与真实模型

调用连接记录应用怎样访问 LiteLLM 或服务商 API，包括协议、地址、能力和本地凭据变量名。TokenPilot 不保存凭据值。真实模型把连接和请求中的模型名称组合起来，成本规则与 AI Unit 换算率都属于真实模型。

### 虚拟模型与分流

虚拟模型是业务代码使用的稳定名称，例如 `fast-text` 或 `deep-reasoning`。它指向真实模型，并设置顺序、权重、时间、用户条件和临时切换。每次发布都会生成固定版本，调用端会回传自己实际使用的版本。

应用可以一直请求同一个虚拟模型，后台发布配置后即可切换服务商或真实模型。图片或语音请求只会选择声明了相应能力的候选模型。

### 数据统计与报表

管理页面可以查看调用、失败后改用记录、Token、延迟、错误、模型花费和 AI Unit。筛选器可以组合内置字段与已定义的自定义字段，保存后可放到当前应用首页。统计读取 ClickHouse；配置和运行状态保存在 PostgreSQL。

### 模型花费、AI Unit 与用户额度

模型花费是每次尝试产生的实际费用。调用端上报金额时直接使用；没有上报时，第一条匹配的成本规则负责计算。AI Unit 使用另一套单价表，用于产品用量和用户额度。修改成本规则不会改变 AI Unit。缺少规则时页面显示“未定价”或“未计算”，不会按零处理。

每个应用用户可以设置 AI Unit 额度。强制限制会在调用前预留估算值，完成后按实际结果结算；请求没有发出时释放预留。拉闸和重置额度都会写入审计记录，并保留历史用量。

## 使用边界

- 消息内容只在发起模型请求的程序内使用，上报不包含提示词和回复正文。
- 服务商密钥保存在运行程序的环境变量或团队已有的密钥系统中。
- SDK 和 Connector 使用本地 SQLite 队列与稳定事件标识重试上报。
- PostgreSQL 与 ClickHouse 都是必需组件，分别承担配置和统计职责。
- 被拒绝的发布不会覆盖最后一次成功应用的配置。

## 建议配置顺序

1. 部署 TokenPilot，完成首次设置。
2. 创建应用和应用接入密钥。
3. 定义允许应用上报的事件字段和用户字段。
4. 添加调用连接和真实模型，按需配置模型花费备用规则，并发布 AI Unit 换算率。
5. 创建虚拟模型，用代表性条件模拟结果后发布。
6. 接入 Node SDK、Python SDK 或 LiteLLM Connector，并验证一次真实调用。
7. 检查用户、尝试链、模型花费、AI Unit 和保存的报表。
8. 先以只统计方式分配用户额度，确认身份与单价覆盖完整后，再启用提醒或强制限制。

先检查第一条真实调用，再开启额度强制限制。缺少的成本或 AI Unit 规则会直接显示，应在用于决策前补齐。
