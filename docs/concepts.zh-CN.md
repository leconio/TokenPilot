# 概念与计算原理

[English](concepts.md)

## 用量事件

每种接入都会为每次模型尝试发送一个事件。事件包含稳定标识、模型身份、实际用量、耗时、结果、分流原因和允许上报的业务字段，不包含提示词或模型回复。重复使用同一个事件标识时，系统返回第一次的结果，不会重复记录。

常见用量类型包括：

- `request`：一次模型尝试；
- `uncached_input_tokens`：没有命中缓存的输入 Token；
- `cache_read_tokens`：从服务商缓存读取的输入 Token；
- `cache_write_tokens`：写入服务商缓存的输入 Token；
- `reasoning_tokens`：支持该能力的服务商上报的推理 Token；
- `output_tokens`：模型可见输出 Token。

## 应用、调用连接、真实模型与虚拟模型

| 名称     | 含义                                                               |
| -------- | ------------------------------------------------------------------ |
| 应用     | 用户、字段、报表、模型配置和密钥的隔离边界                         |
| 调用连接 | 调用端怎样访问 LiteLLM、OpenAI 兼容服务或 Anthropic                |
| 真实模型 | 调用连接加上实际发送的模型名称，成本和 AI Unit 规则绑定在这里      |
| 虚拟模型 | 业务代码使用的稳定名称，负责选择一个或多个真实模型并确定它们的顺序 |

TokenPilot 只保存 `OPENAI_API_KEY` 这样的凭据引用，不保存凭据值。同一个服务商模型如果通过两条连接调用，应建立两条真实模型记录，分别管理分流和统计。

## 服务商成本

调用端上报成本时，TokenPilot 直接使用这笔金额。这样可以记录账号协议、批处理、服务等级、促销等无法用一种固定 Token 价格表达的情况。事件还会记录币种以及金额是否为估算值。

调用端不能提供金额时，TokenPilot 按顺序检查成本规则。规则可以使用内置字段，也可以使用已经定义类型的事件属性和用户属性。第一条匹配规则生效，金额由可选的调用固定金额和已配置的实际用量组成：

```text
规则成本 = 调用固定金额 + Σ（实际用量 × 每单位金额）
```

没有上报金额且没有规则匹配时，这次尝试会标记为未定价。调用端始终上报成本时，规则列表可以为空。上报成本保留自身币种；规则使用应用币种。后续正式结果会写入一条带签名的调整记录，不会修改原记录。

## AI Unit

AI Unit 使用独立单价表。一个简单示例可以是：

```text
1,000,000 输入 Token        = 1.000000 AIU
1,000,000 缓存读取 Token    = 0.250000 AIU
1,000,000 缓存写入 Token    = 0.500000 AIU
1,000,000 推理 Token        = 4.000000 AIU
1,000,000 输出 Token        = 3.000000 AIU
```

内部使用整数微 AIU 保存结果，只在规定边界进行一次舍入，不使用浮点数。每次计算都会保存实际使用的已发布单价快照、舍入方式和尝试计算规则。

服务商成本和 AI Unit 是两套独立规则。修改成本上报或成本规则，不会改变 AI Unit 换算、额度、预留和拉闸。页面会分别显示缺少的成本规则与 AI Unit 规则。

## 虚拟模型与分流

虚拟模型是业务代码使用的稳定模型名称。它内部维护完整的分流设置，包括：

- 一个默认候选；
- 按顺序排列的失败候选；
- 零个或多个时间条件；
- 带明确过期时间的临时规则；
- 单调递增的策略修订号。

SDK 使用 ETag 检查策略更新，并回传自己实际使用的版本。短暂故障时，它会读取本地保存的最后一份有效策略。格式错误或已经过期的策略会被拒绝。

## 额度生命周期

“只统计”模式只记录 AI Unit，不阻止请求。强制限制模式使用下面的过程：

```text
检查 → 预留 → 调用模型 → 结算
                    └──→ 释放
过期预留 ─────────────→ 过期
```

每个状态转换都可以安全重试。用于额度判断的余额和变动记录保存在 PostgreSQL。ClickHouse 只接收用于统计的投影，不参与是否允许请求的决定。

## 数据由谁负责

| 数据                                    | 保存位置                     |
| --------------------------------------- | ---------------------------- |
| 模型目录、成本规则、AI Unit、策略、密钥 | PostgreSQL                   |
| 计算结果、用户额度、预留、变动记录      | PostgreSQL                   |
| 队列租约和短期协调状态                  | Redis                        |
| 报表和仪表盘                            | ClickHouse                   |
| SDK 或 Connector 重试缓冲               | 调用端旁边的本地 SQLite 文件 |

统计不会在 ClickHouse 故障时改读 PostgreSQL。ClickHouse 不可用时，报表直接返回错误，不会改用另一种数据来源。

TokenPilot 不代理模型流量。Node、Python SDK 可以直接调用已注册连接；LiteLLM Connector 则让 LiteLLM 保持在请求链路中。两种方式使用同一份分流策略和同一种用量事件。

## 全新数据库规则

项目仍处于开发阶段。PostgreSQL 和 ClickHouse 都从空卷创建当前结构。发现未知对象或混合结构时会拒绝启动；开发环境应删除对应隔离卷，再创建当前结构。项目不保留旧结构转换逻辑。
